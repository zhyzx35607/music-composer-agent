# 音乐创作智能体项目分工计划

## 1. 项目周期总览

| 阶段 | 时间 | 人数 | 目标 | 完成标准 |
|---|---:|---:|---|---|
| 第一阶段：中期检查版 | 4 天 | 3 人 | 跑通音乐创作平台的核心链路 | 前端可以输入需求，后端可以生成结果，智能体可以输出作曲方案和 caption，Text2MIDI 与 FluidSynth 可以完成 MIDI 和 WAV 输出 |
| 第二阶段：最终完善版 | 5 天 | 4 人 | 完善功能、交互、反馈修改和系统稳定性 | 支持生成、播放、下载、历史版本、反馈修改、修改解释，整体系统可以稳定演示 |

## 2. 总体模块划分

| 模块 | 主要内容 | 对应人员 |
|---|---|---|
| 前端页面模块 | 输入框、参数选择、生成按钮、播放器、历史记录、反馈修改框 | 成员 A |
| 后端服务模块 | API 接口、文件管理、版本管理、前后端联调 | 成员 B |
| 智能体模块 | 中文需求理解、参数补全、作曲方案生成、英文 caption 生成、反馈修改 | 成员 C |
| 生成链路模块 | Text2MIDI 调用、MIDI 文件生成、FluidSynth 转 WAV、SoundFont 配置 | 中期前由成员 C 负责，中期后由成员 D 专门负责 |

## 9. 接口约定表

| 接口 | 方法 | 作用 | 负责人 | 阶段 |
|---|---|---|---|---|
| `/api/generate` | POST | 首次生成音乐 | 成员 B、成员 C | 第一阶段必须完成 |
| `/api/revise` | POST | 根据用户反馈生成新版本 | 成员 B、成员 C | 第二阶段必须完成 |
| `/api/versions` | GET | 获取历史版本列表 | 成员 B | 第二阶段完成 |
| `/api/version/{version_id}` | GET | 获取某个版本详情 | 成员 B | 第二阶段完成 |
| `/outputs/{filename}` | GET | 访问 MIDI 或 WAV 文件 | 成员 B、成员 D | 第一阶段必须完成 |

## 10. `/api/generate` 数据格式

| 项目 | 类型 | 说明 |
|---|---|---|
| `user_prompt` | string | 用户输入的中文音乐需求 |
| `style` | string | 音乐风格，例如 pop ballad、ambient、electronic |
| `mood` | string | 音乐情绪，例如 nostalgic、hopeful、calm |
| `tempo` | string 或 number | 音乐速度，例如 slow、medium、fast 或具体 BPM |
| `instruments` | array | 乐器列表，例如 piano、strings、guitar |

请求示例：

```json
{
  "user_prompt": "我想要一首毕业季的歌，抒情一点",
  "style": "pop ballad",
  "mood": "nostalgic",
  "tempo": "medium",
  "instruments": ["piano", "strings"]
}
```

响应示例：

```json
{
  "version_id": "v1",
  "caption": "A warm nostalgic pop ballad instrumental about graduation farewell, featuring piano and strings, medium tempo, lyrical melody and emotional atmosphere.",
  "midi_url": "/outputs/v1.mid",
  "audio_url": "/outputs/v1.wav",
  "plan": {
    "theme": "graduation farewell",
    "style": "pop ballad",
    "mood": ["nostalgic", "warm"],
    "tempo": 82,
    "key": "C major",
    "instruments": ["piano", "strings"],
    "structure": ["intro", "verse", "chorus", "ending"]
  }
}
```

## 11. `/api/revise` 数据格式

| 项目 | 类型 | 说明 |
|---|---|---|
| `version_id` | string | 要修改的历史版本 ID |
| `feedback` | string | 用户输入的修改意见 |

请求示例：

```json
{
  "version_id": "v1",
  "feedback": "这版太悲伤了，想要更有希望一点，加一点钢琴"
}
```

响应示例：

```json
{
  "version_id": "v2",
  "parent_version_id": "v1",
  "caption": "A brighter and more hopeful pop ballad instrumental about graduation farewell, featuring piano, warm strings and acoustic guitar, with a moderate tempo and uplifting ending.",
  "midi_url": "/outputs/v2.mid",
  "audio_url": "/outputs/v2.wav",
  "plan": {
    "theme": "graduation farewell",
    "style": "pop ballad",
    "mood": ["hopeful", "warm"],
    "tempo": 88,
    "key": "C major",
    "instruments": ["piano", "strings", "acoustic guitar"]
  },
  "change_reason": "根据用户希望音乐更有希望感的反馈，将情绪从偏悲伤调整为温暖和希望，提高速度，并加入更明亮的钢琴和吉他音色。"
}
```

## 12. 详细说明

### 12.1 为什么第一阶段只安排 3 人

第一阶段的重点是中期检查前跑通主流程，不追求功能完整和页面精美。此时 3 个人分工比较合适：

- 成员 A 专注前端页面，保证老师能看到一个完整的平台入口。
- 成员 B 专注后端接口，保证前端、智能体和生成链路可以连接。
- 成员 C 专注智能体和生成模型，保证项目核心能力可以展示。

这一阶段最重要的是“能跑通”，不是“做完美”。

### 12.2 为什么第二阶段增加第 4 人

中期检查后，系统已经有基础链路，接下来工作会变成并行优化：

- 前端需要美化和完善交互。
- 后端需要补全接口和版本管理。
- 智能体需要支持反馈修改和解释。
- Text2MIDI 与 FluidSynth 需要稳定运行。

生成链路比较容易遇到环境、路径、依赖和输出质量问题，因此第二阶段让成员 D 专门负责这部分，可以减少成员 C 的压力，让成员 C 专心体现“智能体”能力。

### 12.3 成员 A 的重点

成员 A 的目标不是做复杂页面，而是做出一个像产品的音乐创作平台。页面建议包含：

- 顶部标题区域
- 中文需求输入框
- 风格、情绪、速度、乐器参数选择
- 生成音乐按钮
- 生成状态提示
- 播放器
- 下载按钮
- 历史版本区域
- 反馈修改输入框
- 智能体作曲方案展示区域

中期前页面可以简单，但最终版需要让用户一眼看懂流程。

### 12.4 成员 B 的重点

成员 B 是系统连接者，重点是接口稳定和数据结构清楚。建议后端内部至少维护这些信息：

- `version_id`
- `parent_version_id`
- `user_prompt`
- `feedback`
- `plan`
- `caption`
- `midi_path`
- `audio_path`
- `change_reason`
- `created_at`

第一阶段可以先用 JSON 文件保存版本数据，没必要一开始就上数据库。最终版如果时间充足，可以再换成 SQLite。

### 12.5 成员 C 的重点

成员 C 是项目亮点的负责人。Text2MIDI 只是底层生成工具，真正体现毕业设计价值的是智能体这一层。

智能体至少要完成四件事：

- 理解中文需求
- 补全音乐参数
- 生成结构化作曲方案
- 生成英文 caption

第二阶段还要重点完成：

- 根据用户反馈修改上一版 plan
- 对比修改前后的参数变化
- 解释为什么这样修改

例如用户反馈“更欢快一点”，智能体不应该只重新生成一句 caption，而应该体现明确修改：

```json
{
  "mood": {
    "from": ["sad", "nostalgic"],
    "to": ["hopeful", "warm"]
  },
  "tempo": {
    "from": 76,
    "to": 88
  },
  "key": {
    "from": "A minor",
    "to": "C major"
  },
  "instruments": {
    "add": ["acoustic guitar", "bright piano"],
    "remove": []
  }
}
```

### 12.6 成员 D 的重点

成员 D 的任务是让生成链路稳定，不让最终演示卡在模型调用或音频转换上。

重点包括：

- Text2MIDI 项目能被后端调用
- 输入 caption 后可以生成 `.mid`
- FluidSynth 可以把 `.mid` 转成 `.wav`
- SoundFont 文件路径固定
- 输出文件路径统一
- 生成失败时有兜底方案
- 写清楚项目启动步骤

如果 Text2MIDI 生成质量不稳定，成员 D 需要准备几个稳定可用的演示 caption 和兜底 MIDI / WAV 文件，保证最终展示时不会失败。

### 12.7 为什么不单独设置测试阶段

本项目时间较短，总共只有 9 天，因此不单独设置测试阶段。测试和调试直接放入每天开发任务中：

- 成员 A 每天验证页面交互是否正常。
- 成员 B 每天验证接口是否能返回正确数据。
- 成员 C 每天验证智能体输出格式是否稳定。
- 成员 D 每天验证 MIDI 和 WAV 是否可以正常生成。

最终第 9 天只做整体检查和演示准备，不再安排大规模新功能。

## 13. 优先级表

| 优先级 | 功能 | 说明 |
|---|---|---|
| P0 | 中文需求生成音乐 | 项目最核心功能，必须完成 |
| P0 | Text2MIDI 生成 MIDI | 没有 MIDI 就没有音乐生成结果 |
| P0 | FluidSynth 转 WAV | 前端播放需要 WAV 音频 |
| P0 | 前端播放器 | 老师检查时必须能听到结果 |
| P1 | 作曲方案展示 | 体现智能体能力 |
| P1 | 历史版本 | 体现多轮创作过程 |
| P1 | 反馈修改 | 体现合作创作能力 |
| P1 | 修改原因解释 | 体现智能体可解释性 |
| P2 | 页面美化 | 提升观感，但不能影响核心功能 |
| P2 | 数据库存储 | 时间充足时再做，可以先用 JSON 文件 |
| P2 | 异步任务队列 | 时间充足时再做，第一版可以同步生成 |

## 14. 风险与处理方式

| 风险 | 影响 | 处理方式 |
|---|---|---|
| Text2MIDI 生成质量不稳定 | 音乐效果可能不好 | 准备稳定 caption 和兜底音频，论文中强调智能体流程 |
| FluidSynth 环境配置失败 | WAV 无法生成 | 提前固定 SoundFont 路径，并整理启动说明 |
| 前后端接口格式不统一 | 联调困难 | 第 1 天就确定请求和响应格式 |
| 反馈修改逻辑复杂 | 第二阶段进度受影响 | 先做规则化修改，再接入大模型优化 |
| 时间不足 | 最终功能不完整 | 优先完成 P0 和 P1，放弃 P2 |

## 16. 总结

本项目的开发重点不是重新训练音乐生成模型，而是完成一个面向中文用户的音乐创作智能体平台。

中期检查前，目标是把链路跑通：

```text
前端输入
→ 后端接口
→ 智能体生成 plan 和 caption
→ Text2MIDI 生成 MIDI
→ FluidSynth 转 WAV
→ 前端播放
```

中期检查后，目标是体现智能体能力：

```text
历史版本
→ 用户反馈
→ 修改作曲方案
→ 生成新 caption
→ 生成新音乐
→ 解释修改原因
```

最终系统应该能够展示一个完整的“人机合作音乐创作”流程。

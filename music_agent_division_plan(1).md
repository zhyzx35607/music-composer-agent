# 音乐创作智能体项目分工计划

总工期：8 天。前 4 天完成中期检查版，后 4 天完成最终完善版。

## 一、前 4 天分工

### 1. 阶段目标

| 阶段 | 时间 | 人数 | 目标 |
|---|---:|---:|---|
| 中期检查版 | 4 天 | 3 人 | 跑通音乐创作平台核心链路，各模块可以连接，功能允许不完善 |

核心链路：

```text
前端输入中文需求
→ 后端接收请求
→ 智能体生成作曲方案
→ 生成英文 caption
→ 调用 Text2MIDI 生成 MIDI
→ 调用 FluidSynth 转 WAV
→ 前端播放结果
```

### 2. 三人模块分工

| 人员 | 负责模块 | 核心任务 | 前 4 天交付内容 |
|---|---|---|---|
| 成员 A | 前端页面 | 搭建音乐创作平台页面 | 输入框、参数选择、生成按钮、播放器、历史版本区域、反馈输入框 |
| 成员 B | 后端服务 | 搭建后端 API，连接前端和生成模块 | `/api/generate` 接口、输出文件目录、版本 ID、静态文件访问 |
| 成员 C | 智能体 + 生成链路 | 完成智能体基础逻辑，跑通 Text2MIDI 和 FluidSynth | 中文需求转 plan，plan 转英文 caption，caption 转 MIDI，MIDI 转 WAV |

### 3. 前 4 天每日安排

| 天数 | 成员 A：前端页面 | 成员 B：后端服务 | 成员 C：智能体 + 生成链路 | 当天目标 |
|---|---|---|---|---|
| 第 1 天 | 搭建页面结构，完成输入框、参数选择、生成按钮 | 搭建后端框架，完成 `/api/generate` 假数据接口 | 跑通 Text2MIDI，确认英文 caption 可以生成 MIDI | 前后端框架建立，生成模型可以单独运行 |
| 第 2 天 | 完成播放器、下载按钮、历史版本区域 | 完成输出目录、静态文件访问、版本 ID 生成 | 跑通 FluidSynth，将 MIDI 转成 WAV | 后端可以管理输出文件，音频可以被前端访问 |
| 第 3 天 | 对接 `/api/generate`，展示后端返回结果 | 对接成员 C 的生成函数，打通完整 API | 完成中文需求到 plan 和 caption 的基础逻辑 | 完成从输入到生成结果的主链路 |
| 第 4 天 | 优化页面布局和基础交互 | 联调前后端，修复接口问题 | 准备演示案例，处理生成失败兜底 | 可以用于中期检查演示 |

### 4. 前 4 天各成员详细职责

| 人员 | 详细职责 |
|---|---|
| 成员 A | 前端页面不用复杂，但要像一个产品。需要包含“描述你想要的音乐”的输入框，风格、情绪、速度、乐器等参数选择，生成音乐按钮，音频播放器，历史版本区域，以及反馈修改输入框。中期前重点是能操作、能展示、能播放。 |
| 成员 B | 后端负责连接整个系统。中期前必须完成 `/api/generate`，接收前端请求，调用智能体和生成模块，返回 `version_id`、`caption`、`midi_url`、`audio_url`、`plan` 等内容。文件输出目录和前端访问路径要统一。 |
| 成员 C | 智能体是毕设重点。中期前要能理解中文需求，补全音乐参数，生成结构化作曲方案，再把方案转成英文 caption。生成链路方面，要能调用 Text2MIDI 输出 `.mid`，再用 FluidSynth 转成 `.wav`。 |

## 二、中期检查要求

### 1. 中期检查功能要求

| 功能 | 是否必须 | 要求 |
|---|---|---|
| 中文需求输入 | 必须 | 用户可以输入“我想要一首毕业季的歌，抒情一点”这类需求 |
| 参数选择 | 必须 | 至少包含风格、情绪、速度、乐器 |
| 智能体生成作曲方案 | 必须 | 输出结构化 plan，体现智能体能力 |
| 英文 caption 生成 | 必须 | 将作曲方案转换为 Text2MIDI 可用的英文描述 |
| Text2MIDI 生成 MIDI | 必须 | caption 输入后可以得到 `.mid` 文件 |
| FluidSynth 转 WAV | 必须 | `.mid` 可以转换为前端可播放的 `.wav` 文件 |
| 前端播放音频 | 必须 | 生成结果能在网页中播放 |
| 历史版本展示 | 建议完成 | 中期可以先展示基础版本列表，不要求复杂切换 |
| 反馈修改 | 可先做入口 | 中期前可以先做反馈框，真正修改逻辑放到后 4 天完善 |

### 2. 中期检查演示流程

| 步骤 | 演示内容 |
|---|---|
| 1 | 打开音乐创作平台页面 |
| 2 | 输入中文需求，例如“我想要一首毕业季的歌，抒情一点” |
| 3 | 选择风格、情绪、速度、乐器 |
| 4 | 点击“生成音乐” |
| 5 | 展示智能体生成的结构化作曲方案 |

### 3. 中期检查通过标准

| 标准 | 说明 |
|---|---|
| 链路可跑通 | 从前端输入到后端生成，再到前端播放，流程可以完整演示 |
| 模块可连接 | 前端、后端、智能体、Text2MIDI、FluidSynth 之间有明确连接 |

### 2. API 要求清单

| 接口 | 方法 | 作用 | 阶段 |
|---|---|---|---|
| `/api/generate` | POST | 首次生成音乐 | 前 4 天必须完成 |
| `/api/revise` | POST | 根据用户反馈生成新版本 | 后 4 天完成 |
| `/api/versions` | GET | 获取历史版本列表 | 后 4 天完成 |
| `/api/version/{version_id}` | GET | 获取某个版本详情 | 后 4 天完成 |
| `/outputs/{filename}` | GET | 访问 MIDI 或 WAV 文件 | 前 4 天必须完成 |

### 3. `/api/generate` 要求

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_prompt` | string | 用户输入的中文音乐需求 |
| `style` | string | 音乐风格，例如 pop ballad、ambient、electronic |
| `mood` | string | 音乐情绪，例如 nostalgic、hopeful、calm |
| `tempo` | string 或 number | 音乐速度，例如 slow、medium、fast 或具体 BPM |
| `instruments` | array | 乐器列表，例如 piano、strings、guitar |

请求示例：

```json
{
  "user_prompt": "我想要一首毕业季的歌，抒情一点",
  "style": "pop ballad",
  "mood": "nostalgic",
  "tempo": "medium",
  "instruments": ["piano", "strings"]
}
```

响应示例：

```json
{
  "version_id": "v1",
  "caption": "A warm nostalgic pop ballad instrumental about graduation farewell, featuring piano and strings, medium tempo, lyrical melody and emotional atmosphere.",
  "midi_url": "/outputs/v1.mid",
  "audio_url": "/outputs/v1.wav",
  "plan": {
    "theme": "graduation farewell",
    "style": "pop ballad",
    "mood": ["nostalgic", "warm"],
    "tempo": 82,
    "key": "C major",
    "instruments": ["piano", "strings"],
    "structure": ["intro", "verse", "chorus", "ending"]
  }
}
```

### 4. `/api/revise` 要求

| 字段 | 类型 | 说明 |
|---|---|---|
| `version_id` | string | 要修改的历史版本 ID |
| `feedback` | string | 用户输入的修改意见 |

请求示例：

```json
{
  "version_id": "v1",
  "feedback": "这版太悲伤了，想要更有希望一点，加一点钢琴"
}
```

响应示例：

```json
{
  "version_id": "v2",
  "parent_version_id": "v1",
  "caption": "A brighter and more hopeful pop ballad instrumental about graduation farewell, featuring piano, warm strings and acoustic guitar, with a moderate tempo and uplifting ending.",
  "midi_url": "/outputs/v2.mid",
  "audio_url": "/outputs/v2.wav",
  "plan": {
    "theme": "graduation farewell",
    "style": "pop ballad",
    "mood": ["hopeful", "warm"],
    "tempo": 88,
    "key": "C major",
    "instruments": ["piano", "strings", "acoustic guitar"]
  },
  "change_reason": "根据用户希望音乐更有希望感的反馈，将情绪从偏悲伤调整为温暖和希望，提高速度，并加入更明亮的钢琴和吉他音色。"
}
```

## 五、程序总流程和难点标注

### 1. 程序流程图

```text
用户输入中文需求
        ↓
前端页面收集参数
        ↓
POST /api/generate
        ↓
后端接收请求
        ↓
音乐创作智能体
        ↓
生成结构化作曲方案 plan
        ↓
生成英文 caption
        ↓
调用 Text2MIDI
        ↓
生成 .mid 文件
        ↓
调用 FluidSynth + SoundFont
        ↓
生成 .wav 文件
        ↓
后端保存版本记录
        ↓
前端播放 / 下载 / 展示历史版本
        ↓
用户输入反馈
        ↓
POST /api/revise
        ↓
智能体修改上一版方案
        ↓
生成下一版音乐
```

### 2. 难点标注

| 难点 | 所在模块 | 影响 | 处理方式 |
|---|---|---|---|
| Text2MIDI 生成质量不稳定 | 生成链路 | 音乐效果可能不好，演示时可能不稳定 | 准备稳定 caption 和兜底音频，论文中强调智能体创作流程 |
| FluidSynth 环境配置 | 生成链路 | MIDI 无法转成 WAV，前端无法播放 | 提前固定 SoundFont 文件路径，整理命令和启动说明 |
| 中文需求到英文 caption | 智能体 | caption 质量直接影响生成结果 | 不直接翻译用户原话，先生成结构化 plan，再生成英文 caption |
| 反馈修改逻辑 | 智能体 | 如果只是重新生成，体现不出合作创作 | 记录上一版 plan，根据反馈修改 mood、tempo、key、instruments 等参数 |
| 版本管理 | 后端服务 | 历史版本容易混乱，文件容易覆盖 | 每次生成独立 `version_id`，保存 parent 版本、plan、caption、MIDI、WAV |
| 前后端接口联调 | 前端 + 后端 | 接口字段不统一会拖慢进度 | 第 1 天固定请求和响应格式，后续按格式开发 |
| 生成耗时 | 后端 + 生成链路 | 用户点击后等待时间较长 | 第一版可以同步生成，前端加 loading；时间充足再做异步任务 |
| 时间不足 | 全部模块 | 最终功能可能做不完 | 优先完成 P0 和 P1，P2 功能可以简化 |
